# Codex task-control policy

- Never create Codex internal subagents or use `spawn_agent`. Keep frontier reasoning, planning, review, and integration in the controller.
- Use exactly `gpt-5.6-sol` for the frontier controller. Default to high reasoning; use medium only for bounded short control work. Before xhigh, run `audit-controller-routing` with `hard_arbitration`, an allowed escalation trigger, and a concrete reason. Use max only for `final_arbitration` with explicit user authority or evidence that xhigh remained unresolved. Never spend Sol xhigh/max on mechanical work.
- Classify delegated work as `repeatable` only with exactly `gpt-5.6-luna` and medium reasoning, or `bounded_reasoning` only with exactly `gpt-5.6-terra` and medium or high reasoning. Never use low reasoning. Keep architecture, contract conflicts, trust decisions, error policy, ambiguous fixtures, review, and integration in the frontier controller.
- Create delegated work only as a user-visible Codex task/thread. Before sending its prompt, register it with `executionSurface: visible_task`, explicit delegation, an economical model class, work-class-compatible reasoning, a concrete quota reason, resolved decisions, exact scope, executable acceptance, forbidden decision boundaries, and an explicit task mode.
- Before creating task shells, plan a schema-v1 parallel batch with each candidate's explicit incremental value, lane, dependencies, conflict domains, review cost/capacity, WIP limits, dirty domains, and implementation worktree identity. Default to at least two valuable safe candidates. A ceremonial second candidate is forbidden; an initial single candidate requires an allowed degradation reason, summary, and evidence.
- Register every selected candidate with `parallelPolicy: batch_v1`, its batch/candidate IDs, and matching title/work/task class. Sync all titles, then prepare one dispatch wave. Send every returned prompt and record each real send; do not continue unrelated controlled work while part of the wave is missing.
- Use `control_only` only when no project code, resource, UI, test, or screenshot runner will change. Use `implementation` for those changes and `visual_implementation` when visual acceptance is required; never use `control_only` to bypass the contract gate.
- Default implementation registration to `adaptive_brief` with `scopePolicy: bounded_incidental`. Scope states the objective and functional boundary, not a file whitelist. The worker explores the real owner and chooses relevant validation. Use a schema-v3 hard contract with `strict_scope` only for high-risk irreversible work, shared conflicts, explicit parallel coordination, or explicit user authority.
- An adaptive worker may keep a small same-objective, same-functional-domain repair in the current task only after a structured incidental-repair receipt proves the change is local, reversible, inside registered conflict domains, outside every forbidden/product/architecture/safety/economy/save/external/dependency decision, and backed by RED/GREEN evidence. The final result manifest must disclose every repaired file. Otherwise stop and return to the direct controller.
- For parallel implementation/visual work, result manifests and local artifacts stay in the exact registered worktree. Completion verifies worktree real path, Git repository, branch, base/last-sync ancestry, and candidate HEAD. Never copy a worktree manifest into main merely to satisfy the result protocol.
- If an older task-control tooling/contract bug stopped an already-finished registered-worktree candidate, only the direct controller may use the versioned completion-only recovery with a real host receipt. Preserve the old failure, freeze the same candidate/manifest, forbid business reruns, and do not charge mechanical retry/replacement/fuse.
- Register with a semantic title, apply the returned `set_thread_title` action, and record title sync before sending any work. Never leave a task named `等待主控登记`.
- Keep the semantic title stable and update only its lifecycle prefix. Use the ledger-assigned hierarchical display key to show sibling and parent/child relationships.
- Visible sibling and nested tasks are allowed only when each remains independently visible and registered. Recompute fan-out after completion, failure, reclaim, review, integration, and batch closeout. If code plus independent QA/no-code/readonly work is eligible, do not collapse the batch to one code task.
- Route every later controller-to-worker message through task-control. Ordinary follow-up, clarification, evidence, and notification messages must remain `deferred_local` while the target turn is running or unknown; release only after a real idle observation and record the matching host receipt. Never interrupt for ordinary work. Only explicitly authorized stop/cancel may use the returned steer action, and never send a deferred message after the task leaves `executing`.
- A child starts with `query-parent --context-mode preload`: it receives only the registered direct parent and any verified confirmed `always` checkpoint facts. Missing checkpoint data is non-blocking. Never fork the full controller conversation into an ordinary parallel worker.
- When a prior attempt, unexpected failure, historical route, or ambiguous earlier decision makes parent history materially useful, a child may call `query-parent-context --reason ...` and execute only the returned direct-parent `read_thread` action. Read completed summaries/messages without tool output, stop when older pages become repetitive or irrelevant, and treat raw parent history as advisory. It cannot override project rules, the ledger, current scope, forbidden decisions, or checkpoint authority.
- Otherwise a child may query only its own record, emit progress, emit a bounded incidental-repair receipt, emit a failed/blocked event, emit completion, or emit a notification-failure receipt. Hard-contract failure authority remains controller-fixed; adaptive workers report actual evidence. Pending artifacts never complete central state and the controller ingests them in order.
- A child must notify only its registered direct parent. If the host lacks direct-parent reading or a new decision is required, use the queued clarification path instead of reading ancestors or widening scope.
- Every new implementation uses result protocol v2. The worker completes with a schema-v2 result manifest containing actual files, reasons, validation rationale, and artifacts. Visual work keeps only high-value reference/before/intermediate/after/failure images and cannot complete with a missing, empty, outside-root, or undecodable required artifact. Non-visual work may omit screenshots only with a concrete reason and readable test or before/after results.
- Only the controller may register tasks, ingest events, request changes, explicitly dispatch one mechanical rework, reclaim work, accept, or integrate. New implementation integration must verify the candidate is a Git ancestor of the declared target (default `HEAD`); legacy records without proof remain visibly unverified.
- The direct controller records an acceptance/rejection reason and selects display screenshots. Preserve every attempt; render rejected, reclaimed, and blocked evidence as failed. Refresh the deterministic task-control delivery HTML after real events/reviews, but never keep or rearm a heartbeat merely because the report is stale.
- Observability must piggyback existing lifecycle writes; never create an extra worker turn or progress event for timing. Use the default lean HTML for ordinary closeout. Only when the user asks for timing, consumption, or real-concurrency diagnosis may the controller generate the separate on-demand diagnostic HTML and explicitly supply local OTel/Desktop evidence. Crop diagnostic analysis to dispatch-through-execution-end, show pre/post task time as excluded task-external idle, and calculate the unknown ratio only inside paired active turns. The generated page uses Chinese explanations, 万/亿 human-readable values with exact counts, and task-relative comparison bars. Tokens are already-observed cumulative processing rather than OTel overhead or a quota bill, account snapshots are not per-task bills, and unknown time remains unassigned.
- Treat `changes_requested` as stopped and undecided. Preparing a mechanical rework must not increment attempt or mark it running; only a real host send receipt may confirm the next attempt. Cancel failed sends and use the dedicated zombie-attempt recovery instead of ordinary gates.
- Audit stalled active tasks even when no completion exists. Keep stable objective identity across replacements; after two failed replacements or an exhausted budget, stop new dispatch and show an incident summary. A diagnostic cannot block a milestone without player-impact, normal-lifecycle, growth-trend, and why-blocking evidence.
- Reclaim/block requires a user-visible summary, sent-notification acknowledgement, and refreshed delivery report before replacement. If context health is `handoff_required`, migrate to a clean controller before registering or dispatching more workers.
- Use one heartbeat protocol-v3 automation for the whole batch. Progress renews the logical ledger lease only while the physical one-shot remains untriggered; a consumed one-shot must be replaced after real progress. Heartbeat cleanup is separate from business lifecycle and must not disable reclaim or recovery.
- End every controller event/review/closeout/thread-action pass with `controller-finalize-cycle` and execute its bounded heartbeat action. Keep warnings visible, but do not make heartbeat confirmation a prerequisite for failure ingestion, reclaim, recovery, or unrelated registration. A partial parallel dispatch wave remains fail-closed.
- Every scheduled scan must carry generation, automation ID, action ID, RRULE snapshot, occurrence, and fired-at time. Stale/wrong-ID/expired/repeated invocations may only verify their own snapshot and apply the returned bounded cleanup; they must not process project queues or rearm. Two unchanged business fingerprints stop automatic rearm. Permit one automatic delete compensation, then require manual cleanup, notify once, and resume only with `controller-resume-watchdog --reason ...` after the stalled state is reviewed.
- The controller heartbeat applies only pending, currently actionable title and archive actions. A recorded tool failure becomes audit debt and must not keep the heartbeat or retry automatically; only the registered direct controller may explicitly requeue it. Archive `integrated`, `blocked`, or `reclaimed` descendants before their parent, but retain all ledger history.
- After install or upgrade, run `audit-model-routing`, `audit-thinking-routing`, and `audit-archive-backlog`. Treat active legacy low-thinking tasks as direct-controller reclaim work and missing legacy archive metadata as pending cleanup in memory; only the recorded direct controller may apply actions and record their results.
- Project-local `AGENTS.md`, workflows, tests, and acceptance rules remain authoritative.
- If a visible task is not registered, stop and report that it is waiting for controller registration.
