# Codex task-control policy

- Never create Codex internal subagents or use `spawn_agent`. Keep frontier reasoning, planning, review, and integration in the controller.
- Use exactly `gpt-5.6-sol` for the frontier controller. Default to high reasoning; use medium only for bounded short control work. Before xhigh, run `audit-controller-routing` with `hard_arbitration`, an allowed escalation trigger, and a concrete reason. Use max only for `final_arbitration` with explicit user authority or evidence that xhigh remained unresolved. Never spend Sol xhigh/max on mechanical work.
- Classify delegated work as `repeatable` only with exactly `gpt-5.6-luna` and medium reasoning, or `bounded_reasoning` only with exactly `gpt-5.6-terra` and medium or high reasoning. Never use low reasoning. Keep architecture, contract conflicts, trust decisions, error policy, ambiguous fixtures, review, and integration in the frontier controller.
- Create delegated work only as a user-visible Codex task/thread. Before sending its prompt, register it with `executionSurface: visible_task`, explicit delegation, an economical model class, work-class-compatible reasoning, a concrete quota reason, resolved decisions, exact scope, executable acceptance, forbidden decision boundaries, and an explicit task mode.
- Before creating task shells, plan a schema-v1 parallel batch with candidate lanes, dependencies, conflict domains, review cost/capacity, WIP limits, dirty domains, and implementation worktree identity. Default to at least two safe candidates. A single candidate requires an allowed degradation reason, concrete summary, and evidence.
- Register every selected candidate with `parallelPolicy: batch_v1`, its batch/candidate IDs, and matching title/work/task class. Sync all titles, then prepare one dispatch wave. Send every returned prompt and record each real send; do not continue unrelated controlled work while part of the wave is missing.
- Use `control_only` only when no project code, resource, UI, test, or screenshot runner will change. Use `implementation` for those changes and `visual_implementation` when visual acceptance is required; never use `control_only` to bypass the contract gate.
- Before implementation registration, create a schema-v2 JSON contract inside the project root with a revision/commit, explicit allowed write paths, reuse requirements, forbidden new paths/reimplementations, ordered stage gates, named evidence commands, controller-fixed error policy, and a visual oracle when applicable. Existing schema-v1 ledger tasks may finish their bound attempt, but new registration requires v2.
- Register with a semantic title, apply the returned `set_thread_title` action, and record title sync before sending any work. Never leave a task named `等待主控登记`.
- Keep the semantic title stable and update only its lifecycle prefix. Use the ledger-assigned hierarchical display key to show sibling and parent/child relationships.
- Visible sibling and nested tasks are allowed only when each remains independently visible and registered. Recompute fan-out after completion, failure, reclaim, review, integration, and batch closeout. If code plus independent QA/no-code/readonly work is eligible, do not collapse the batch to one code task.
- Route every later controller-to-worker message through task-control. Ordinary follow-up, clarification, evidence, and notification messages must remain `deferred_local` while the target turn is running or unknown; release only after a real idle observation and record the matching host receipt. Never interrupt for ordinary work. Only explicitly authorized stop/cancel may use the returned steer action, and never send a deferred message after the task leaves `executing`.
- A child task may query only its own record, emit staged progress, emit a first-class failed/blocked event, emit completion, or emit a notification-failure receipt. Failure may occur before required stages finish but must bind the attempted stage, classified cause, command summary, and registered evidence.
- A child must resolve its direct parent from the ledger and notify only that parent.
- Every newly registered implementation contract must fix result requirements. The worker completes with a schema-v1 result manifest; visual work keeps only high-value reference/before/intermediate/after/failure images and cannot complete with a missing, empty, outside-root, or undecodable required artifact. Non-visual work may omit screenshots only with a concrete reason and readable test or before/after results.
- Only the controller may register tasks, ingest events, request changes, explicitly dispatch one mechanical rework, reclaim work, accept, or integrate.
- The direct controller records an acceptance/rejection reason and selects display screenshots. Preserve every attempt; render rejected, reclaimed, and blocked evidence as failed. Refresh the deterministic task-control delivery HTML after real events/reviews, but never keep or rearm a heartbeat merely because the report is stale.
- Observability must piggyback existing lifecycle writes; never create an extra worker turn or progress event for timing. Use the default lean HTML for ordinary closeout. Only when the user asks for timing, consumption, or real-concurrency diagnosis may the controller generate the separate on-demand diagnostic HTML and explicitly supply local OTel/Desktop evidence. The generated page uses Chinese explanations, 万/亿 human-readable values with exact counts, and task-relative comparison bars. Tokens are already-observed cumulative processing rather than OTel overhead or a quota bill, account snapshots are not per-task bills, and unknown time remains unassigned.
- Treat `changes_requested` as stopped and undecided, not as running. Reclaim comprehension, judgment, and missing-spec failures immediately; after a second failed review, reclaim instead of looping.
- Audit stalled active tasks even when no completion exists. Keep stable objective identity across replacements; after two failed replacements or an exhausted budget, stop new dispatch and show an incident summary. A diagnostic cannot block a milestone without player-impact, normal-lifecycle, growth-trend, and why-blocking evidence.
- Reclaim/block requires a user-visible summary, sent-notification acknowledgement, and refreshed delivery report before replacement. If context health is `handoff_required`, migrate to a clean controller before registering or dispatching more workers.
- Use exactly one heartbeat protocol-v2 automation for the entire direct-controller batch. A prepared dispatch wave may start the lease before the first real send so a partial wave remains recoverable. Merely planned batches and unresolved single-candidate degradation do not keep a heartbeat. A returned action is prepared, not committed: create a new `COUNT=1` automation, confirm its new ID, switch the ledger, then delete the retired ID.
- End every controller event/review/closeout/thread-action pass with `controller-finalize-cycle`, execute and confirm its heartbeat action, then require `controller-assert-business-ready` before resuming project work. A terminal controller with an unconfirmed create must compare-delete that action ID/generation and delete the old confirmed automation; do not continue while either deletion is unconfirmed.
- Every scheduled scan must carry generation, automation ID, action ID, RRULE snapshot, occurrence, and fired-at time. Stale/wrong-ID/expired/repeated invocations may only verify their own snapshot and apply `delete_stale_automation`; they must not process project queues or rearm. Record delete failures, fuse after three, and notify once.
- The controller heartbeat applies only pending, currently actionable title and archive actions. A recorded tool failure becomes audit debt and must not keep the heartbeat or retry automatically; only the registered direct controller may explicitly requeue it. Archive `integrated`, `blocked`, or `reclaimed` descendants before their parent, but retain all ledger history.
- After install or upgrade, run `audit-model-routing`, `audit-thinking-routing`, and `audit-archive-backlog`. Treat active legacy low-thinking tasks as direct-controller reclaim work and missing legacy archive metadata as pending cleanup in memory; only the recorded direct controller may apply actions and record their results.
- Project-local `AGENTS.md`, workflows, tests, and acceptance rules remain authoritative.
- If a visible task is not registered, stop and report that it is waiting for controller registration.
