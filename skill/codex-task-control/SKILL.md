# Codex Task Control

Use the bundled `scripts/task-control.mjs` as the single task-lifecycle and model-routing control plane. It owns visible-task identity, routing, lifecycle evidence, review state, and archive coordination. Project authority remains limited to the project's own development rules, business workflows, commands, tests, and acceptance constraints.

## Workflow

1. Resolve the project root and read its `AGENTS.md` plus the applicable SOP/workflow sources before doing project work.
2. Use the optional project adapter only to locate project rules, applicable business workflows, and project-specific acceptance constraints. Never require or accept a project model-routing source: model, thinking, work-class, and lifecycle policy belong to this user-level skill. Never copy policy text into the adapter or let it override project rules.
3. Set `CODEX_HOME` (or pass `--codex-home`) so the ledger is stored at `$CODEX_HOME/task-control/`; use `--task-control-home` only when passing the exact task-control root. The index is `projects.json`; each project has an isolated `projects/<project-key>/task-registry.json` and `events/` directory.
4. Do the work in the controller by default. Never call internal subagent, multi-agent, or `spawn_agent` capabilities. Delegated work must exist as a user-visible Codex task/thread in the sidebar.
5. Classify work before creating a task shell. Use `repeatable` only when rules, inputs, expected results, and the test oracle are explicit; it must use exactly `gpt-5.6-luna` with medium thinking. Use `bounded_reasoning` only after contracts and policy decisions are resolved but local code understanding is still needed; it must use exactly `gpt-5.6-terra` with medium thinking by default or high thinking when the decided work spans enough code to justify it. Never use low thinking. Keep work in the `gpt-5.6-sol` controller whenever architecture, contract conflict, trusted-source selection, error policy, fixture validity, or another judgment remains unresolved. Route bounded short controller work as `bounded_control` with medium thinking and use `frontier_control` with high thinking by default. Before selecting xhigh, run `audit-controller-routing` with `hard_arbitration`, an allowed escalation trigger, and a concrete reason. Use max only as `final_arbitration` with `user_explicit` or `xhigh_unresolved` authority. Never spend Sol xhigh/max on search, formatting, commands, repeated tests, or mechanical implementation.
6. Authorize a worker only when the expected premium-quota saving exceeds the new context and coordination overhead. A model upgrade never repairs an ambiguous task; resolve decisions and narrow the contract first.
7. Classify every new registration with an explicit `--task-mode`. Use `control_only` only for ledger, audit, inspection, or other work that does not modify project code, resources, UI, tests, or screenshot runners. Use `implementation` for code/resource/test changes and `visual_implementation` when visual output also needs a fixed oracle. Never use `control_only` to bypass an implementation contract. Existing records without this field migrate in memory as `legacy_unclassified`; do not rewrite them merely to silence the distinction.
8. Before registering an implementation task, create a project-owned JSON contract inside the project root. Bind it with `--implementation-contract`; include schema version 1, `contractRevision` or `contractCommit`, non-empty `reuseRequirements`, `forbiddenNewPaths`, `forbiddenReimplementations`, ordered `stageGates`, named `evidenceCommands`, and controller-fixed `errorPolicy`. A visual task additionally requires `visualOracle`. The skill stores the normalized snapshot and SHA-256 digest, then rechecks it at dispatch, progress, and completion. Any drift fails closed; the worker cannot change error policy or acceptance by editing the manifest.
9. Create a visible task shell only to obtain its thread ID. Register it with a semantic `--title`, never the placeholder `等待主控登记`. Require `--execution-surface visible_task`, `--delegation explicit`, `--model-class economical`, work-class-compatible medium or high thinking, a concrete `--quota-reason`, `--work-class`, `--decision-status resolved`, exact `--scope`, executable `--acceptance`, explicit `--forbidden-decisions`, and the chosen task mode/contract.
10. Read the registration result's `requiredThreadActions`. Call `codex_app__set_thread_title` with the exact returned title, then record success with `controller-record-title-synced`. Do not send the work prompt until `dispatchAllowed: true`. On tool failure, call `controller-record-title-failed` and stop dispatch. After the work prompt is really sent, call `controller-record-dispatched`; never record dispatch before a successful send. This call rechecks the contract digest and starts the adaptive lease. Apply its returned `heartbeatAction` immediately.
11. Keep the semantic title stable. The ledger allocates `01`, `02`, and nested `01.1` display keys and changes only the lifecycle prefix: `执行`, `待审`, `待决`, `返工`, `收回`, `接收`, `完成`, or `阻塞`. `changes_requested` means stopped and awaiting a controller routing decision; it never means the worker is running.
12. Visible tasks may be siblings or nested visible tasks. Every visible task must be independently registered and remain directly inspectable by the user.
13. A worker may only query itself, create a progress event, create a completion event, or create a notification-failure receipt. Use `progress` only for a real checkpoint, not repetitive logs. For implementation work, every progress event must name a registered stage and provide all required `id=reference` evidence; required stages run in contract order and are tracked per attempt. The worker must obtain `parentThreadId` and `notificationText` from the ledger and notify only that direct parent.
14. The controller alone ingests events and marks `changes_requested`, dispatches rework, reclaims work, blocks, accepts, and integrates. Implementation completion fails until every required stage has been ingested for the current attempt. Completion and review output return task mode, contract version/digest, completed stages, and missing stages. Classify each review failure as `mechanical`, `comprehension`, `judgment`, or `spec_missing`. Allow the same worker one explicit rework only for `mechanical`; rework starts a new stage attempt against the same fixed contract. Reclaim all other failures immediately, and reclaim after a second failed review. A Luna-to-Terra escalation must close the old task as `reclaimed` and register a new bounded visible task.
15. Use heartbeat protocol v2 and exactly one confirmed one-shot automation per direct controller. A ledger rearm now only prepares `create_controller_heartbeat` or `delete_controller_heartbeat`; it does not advance the confirmed generation. For create, call Codex App automation create with `COUNT=1`, a name/prompt containing the returned `actionId` and generation, and a 30-second host deadline. Always create a new automation ID; never update the confirmed heartbeat in place. After App success, call `controller-confirm-heartbeat-action --action-id ... --automation-id <new-id>` to atomically switch the ledger, then apply the returned `delete_retired_automation` for the old ID. For delete, remove the exact confirmed ID and confirm the pending delete. On error or timeout, call `controller-record-heartbeat-action-failed`; the old confirmed generation remains authoritative.
16. Include `--heartbeat-generation`, resolved `--automation-id`, `--heartbeat-action-id`, quoted `--heartbeat-rrule`, `--heartbeat-occurrence`, and `--heartbeat-fired-at` in every scheduled scan. If create succeeded but its response was lost, the new automation resolves its own ID by finding the unique automation file containing `actionId`; a scan of the pending generation returns `confirm_observed_heartbeat`, allowing safe adoption. If a pending action exceeded 30 seconds, the old confirmed automation returns `compensate_timed_out_heartbeat_action`; record the failure before scanning business queues.
17. Treat a stale, wrong-ID, expired, `COUNT!=1`, or second occurrence as a host-cleanup path, not a silent no-op and not a business scan. `controller-scan-events` returns empty work queues plus `heartbeatAction.type=delete_stale_automation`, restricted to the triggering automation ID and snapshot generation. Verify the local automation snapshot still names that generation, then delete only that automation with a 30-second deadline. Record deletion failure. Three consecutive stale triggers or three delete failures open the fuse, persist evidence, and return one notification; after a real notification, call `controller-mark-heartbeat-notification-sent`. Never rearm from a stale invocation. For a current valid scan, reconcile events/thread actions first, then call `controller-rearm-heartbeat --reason reconcile` once and apply its prepared action through the same confirm/fail protocol.
18. Archive only `integrated`, `blocked`, or `reclaimed` tasks. Archive visible descendants first, then their parent. Keep registry and event history; archive only the Codex sidebar thread.
19. After installation or upgrade, run `audit-model-routing`, `audit-thinking-routing`, and `audit-archive-backlog`. The routing audits are read-only; active legacy low-thinking tasks must be reclaimed by their registered direct controller rather than mutated in place. The archive audit groups terminal debt by registered direct controller and labels each item `actionable`, `title_failed`, `archive_failed`, or `waiting_descendants`. Each owner applies only returned actions descendant-first and records success or failure; failed debt stays visible in audit without keeping a heartbeat. Never impersonate another controller or mutate legacy evidence directly.
20. Delete the heartbeat when the returned `heartbeatAction.type` is `delete_controller_heartbeat`, which means no dispatched active work, pending review, routing decision, or currently actionable thread action remains. Historical failure debt does not prevent deletion.

## Safety boundaries

- Do not write `$CODEX_HOME/AGENTS.md`, `$CODEX_HOME/skills/`, project files, or live project ledgers while building or reviewing a candidate.
- Keep project-specific rules in project sources; this skill is an outer control-plane ledger, not a replacement workflow.
- Treat duplicate self-thread IDs across projects, stale events, wrong parents/projects, cycles, invalid enums, and contradictory lifecycle records as errors.
- Reject unsafe thread identifiers before registry reads or event path construction; never interpolate `/`, `\\`, `.`, `..`, or whitespace into an events path.
- Keep provider calls at zero for ledger-only work; this skill records local control evidence and does not invoke a model provider.
- Keep Sol at high by default. Require `audit-controller-routing` evidence before xhigh/max; reject low/none, mechanical escalation, unsupported triggers, missing reasons, and max without explicit authority.
- Forbid internal subagents. Reject any execution surface other than `visible_task`, missing explicit authorization, non-economical workers, low thinking, and missing quota justification.
- Fail closed on a `workClass`/model/thinking mismatch, missing task mode, implementation work without a complete in-project contract, contract digest drift, missing stage evidence, incomplete required stages, unresolved decisions, missing scope/acceptance/decision boundaries, controller-only work, non-mechanical redispatch, a second rework, placeholder titles, stale title acknowledgements, partial title/archive metadata, archive-before-terminal, archive-before-title-sync, and parent archive while a visible descendant remains unarchived.
- Never send a work prompt while `dispatchAllowed` is false, and never call `controller-record-dispatched` until the send succeeds. A failed title/archive action is deferred audit debt and must not be re-emitted or keep a heartbeat; only the registered direct controller may explicitly move it back to `pending` with a recorded reason.
- Never use a repeating heartbeat schedule or in-place heartbeat replacement. Prepare locally, create a new `COUNT=1` automation, confirm its new ID, switch the ledger, and delete the retired ID. A host timeout records failure and preserves the previous confirmed generation.
- Never keep a stale invocation silent. Resolve its own automation ID, verify its stored prompt still carries the triggering generation, and execute only the returned bounded `delete_stale_automation`; do not read project context, process queues, or rearm. Treat inability to compare-and-delete atomically as a Codex App limitation and prefer a future host-native stale cleanup hook.
- Never delete terminal task history. `codex_app__set_thread_archived` cleans the sidebar; the ledger retains identity, lifecycle, notification, title-sync, and archive evidence.
- Never treat a completion file as proof of notification. `notificationStatus: sent` requires a real successful message to the registered direct parent.
- Use atomic registry replacement and exclusive event-file creation. Lock files contain `pid`, `createdAt`, and `nonce`; stale recovery requires a threshold check plus repeated owner verification, and release deletes only a matching nonce.
- Run tests with a temporary `--codex-home` or `--task-control-home`; omitted storage arguments must resolve through a test-only `CODEX_HOME` sandbox.
