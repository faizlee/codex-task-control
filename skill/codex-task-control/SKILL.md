# Codex Task Control

Use the bundled `scripts/task-control.mjs` as the single task-lifecycle and model-routing control plane. It owns visible-task identity, routing, lifecycle evidence, review state, and archive coordination. Project authority remains limited to the project's own development rules, business workflows, commands, tests, and acceptance constraints.

## Workflow

1. Resolve the project root and read its `AGENTS.md` plus the applicable SOP/workflow sources before doing project work.
2. Use the optional project adapter only to locate project rules, applicable business workflows, and project-specific acceptance constraints. Never require or accept a project model-routing source: model, thinking, work-class, and lifecycle policy belong to this user-level skill. Never copy policy text into the adapter or let it override project rules.
3. Set `CODEX_HOME` (or pass `--codex-home`) so the ledger is stored at `$CODEX_HOME/task-control/`; use `--task-control-home` only when passing the exact task-control root. The index is `projects.json`; each project has an isolated `projects/<project-key>/task-registry.json` and `events/` directory.
4. Do the work in the controller by default. Never call internal subagent, multi-agent, or `spawn_agent` capabilities. Delegated work must exist as a user-visible Codex task/thread in the sidebar.
5. Classify work before creating a task shell. Use `repeatable` only when rules, inputs, expected results, and the test oracle are explicit; it must use exactly `gpt-5.6-luna` with medium thinking. Use `bounded_reasoning` only after contracts and policy decisions are resolved but local code understanding is still needed; it must use exactly `gpt-5.6-terra` with medium thinking by default or high thinking when the decided work spans enough code to justify it. Never use low thinking. Keep work in the `gpt-5.6-sol` controller whenever architecture, contract conflict, trusted-source selection, error policy, fixture validity, or another judgment remains unresolved. Route bounded short controller work as `bounded_control` with medium thinking and use `frontier_control` with high thinking by default. Before selecting xhigh, run `audit-controller-routing` with `hard_arbitration`, an allowed escalation trigger, and a concrete reason. Use max only as `final_arbitration` with `user_explicit` or `xhigh_unresolved` authority. Never spend Sol xhigh/max on search, formatting, commands, repeated tests, or mechanical implementation.
6. Authorize a worker only when the expected premium-quota saving exceeds the new context and coordination overhead. Resolve genuine product, safety, architecture, and external-impact decisions first, but do not pre-decide ordinary code navigation, implementation paths, or validation tools that the worker can discover from the real project.
7. Classify every new registration with an explicit `--task-mode`. Use `control_only` only for work that does not modify project code, resources, UI, tests, or runners. Use `implementation` for code/resource/test changes and `visual_implementation` for user-visible output. Existing records without this field migrate in memory as `legacy_unclassified`.
8. Default implementation work to `--execution-policy adaptive_brief`. Registration fields form a schema-v1 brief; optional `--implementation-brief` may add exploration and validation hints, but path whitelists, fixed implementation routes, or mandatory named validators are not allowed in the brief. The worker first inspects the real call chain, then chooses the coherent implementation path and validation method. Completion uses a schema-v2 result manifest with actual affected files, reasons, chosen validation, and evidence relevance. Use headless evidence for logic and GUI/MCP/interactive evidence for UI.
   Upgrade to `hard_contract` only for `high_risk_irreversible`, `shared_conflict`, `parallel_coordination`, or `user_explicit`, with a concrete reason and schema-v3 contract. Hard contracts may freeze true safety/conflict boundaries, but must allow additional and alternative evidence, may not make one validator conclusive, and may not require headless execution to observe GUI/screenshots. Old schema-v1/v2 tasks remain readable and may finish; they are not the default for new work.
9. Before creating task shells, write a project-relative schema-v1 parallel batch manifest and run `controller-plan-parallel-batch`. Record candidates, lanes, dependencies, conflict domains, review cost/capacity, WIP limits, dirty conflict domains, and worktree identity for implementation candidates. Default to at least two independent candidates with distinct incremental value. If the initial wave has only one valuable safe candidate, require a typed `degradationReceipt`; do not add a ceremonial second task. After a real wave starts, completed/blocked/reclaimed peers may naturally shrink the batch to one remaining candidate without a new degradation receipt.
10. Create every task shell needed by the selected fan-out wave only to obtain thread IDs. Register each with `--parallel-policy batch_v1 --batch-id ... --candidate-id ...`, a semantic `--title`, and the existing routing/contract fields. After all selected candidates are registered and their titles are synced, run `controller-prepare-parallel-dispatch`. It atomically records one dispatch wave and returns every required task. Send each real prompt, then call `controller-record-dispatched` for that task. Until every wave member is recorded, unrelated controlled business fails closed and the batch heartbeat keeps the controller accountable. Host message sends cannot be atomic, so an interrupted partial wave remains visible and recoverable rather than being reported as complete.
11. Keep the semantic title stable. The ledger allocates `01`, `02`, and nested `01.1` display keys and changes only the lifecycle prefix: `执行`, `待审`, `待决`, `返工`, `收回`, `接收`, `完成`, or `阻塞`. `changes_requested` means stopped and awaiting a controller routing decision; it never means the worker is running.
12. Visible tasks may be siblings or nested visible tasks. Every visible task must be independently registered and remain directly inspectable by the user. After completion, failure, reclaim, acceptance, integration, or batch closeout, recompute fan-out. When an implementation candidate and a non-conflicting QA/no-code/readonly candidate are both eligible, single-task dispatch is forbidden. Dynamic WIP limits simultaneous review cost and conflict domains, not the total historical task count. Unintegrated candidates cannot satisfy another candidate's dependency.
   Route every later controller-to-worker message through `controller-prepare-message`. Ordinary `follow_up`, `clarification`, `evidence_request`, and `notification` messages use `delivery-mode queue`. When the target is `running` or `unknown`, the command persists `deferred_local` and returns no host send action; do not call `send_message_to_thread`. Re-check the real task turn state and use `controller-release-message` only after it is `idle`; then execute the returned `send_thread_message` and record the matching action ID plus real host receipt. If the task has left `executing`, release cancels the message instead of restarting it. Only `stop` or `cancel` with `user_explicit` or `controller_safety` authority may use `delivery-mode interrupt` and the returned steer action. Never treat the current App setting or a successful generic send call as proof that a running-task message was queued: the current programmatic tool exposes neither an explicit queue mode nor a queue acknowledgement.
13. A worker may only query itself, create progress/completion, submit `report-failure`, or create a notification-failure receipt. Adaptive work reports the actual attempted stage and evidence directly; progress may use worker-chosen labels. Hard-contract evidence remains recoverable or advisory until viable alternative evidence routes are exhausted; one failed validator cannot decide the outcome alone. Once a current attempt has a real dispatch receipt, later lifecycle-title debt cannot suppress its artifacts.
14. The controller alone ingests events and changes lifecycle. A single mechanical retry still requires a real host receipt. Hard-contract predecessor stages may carry forward; adaptive work carries no invented fixed-stage authority. Only an existing schema-v3 hard contract may be amended, and its safety/conflict boundary cannot widen.
15. Piggyback observability on existing lifecycle commands; never add a worker turn merely to measure time. Lean HTML reads only the ledger. Diagnostic HTML is user-requested, local, and may read matching rollout/OTel/Desktop evidence. The report starts with an executive summary that separately counts verified business deliveries, implementation candidates, and control-review results. `control_only` terminal work is “控制审查已通过”, never a business integration. If verified deliverables are zero, say so explicitly. Keep fixed explanations Chinese, use 万/亿 plus exact values, and never assign unknown time or account quota snapshots to one task.
   After a real completion, failure, reclaim, block, or review action, refresh only the lean delivery report. It includes objective runtime/fuse state, failure history, diagnostic value decisions, and incident closeout. `reportNeedsRefresh` alone never keeps heartbeat alive, but an incomplete reclaim/block closeout does. Schema-v2 context-health receipts from `$codex-time-diagnostics` may report `healthy`, `checkpoint_recommended`, or `handoff_recommended`; all are advisory and never block dispatch. Legacy schema-v1 `handoff_required` remains readable and blocking until explicitly migrated.
16. Seal a schema-v1 conversation checkpoint only at a meaningful milestone, before an approved handoff, or when the user asks for one; never add a worker turn just to checkpoint. Store 1-12 concise points plus typed source references, not raw prompts, responses, tool output, or project content. `always` preload is restricted to `user_confirmed`, `project_fact`, `controller_decision`, and `accepted_result`. Candidate, failure, unverified, dispute, and superseded evidence stays on demand. Historical checkpoint files are immutable; the registry stores only the latest verified pointer.
17. A controller handoff is explicit and three-state: `prepared`, `accepted`, or `cancelled`. `controller-prepare-handoff` requires the latest sealed checkpoint and fails closed while any direct task, review/routing/closeout debt, thread action, parallel batch, deferred/prepared controller message, or heartbeat action/automation remains. A prepared handoff blocks new source registration/dispatch but does not retain a heartbeat. The source may cancel it if the successor never accepts. Acceptance verifies the checkpoint digest again, adds the new visible task as a root controller, returns only the confirmed preload points, and permanently retires the old controller from new dispatch. v1 never transfers active child ownership.
18. Use one confirmed `COUNT=1` heartbeat automation per direct controller. Progress, authoritative failure, and finalization renew only the logical lease and reuse the confirmed physical one-shot while it is untriggered; only a fired, deleted, or consumed one-shot may create a new generation. Fingerprint the business lifecycle on every valid watchdog scan. Two consecutive cycles without a task, event, review, message, or batch change disable automatic rearm and request one user-visible notification. Heartbeat state and host automation are control-plane observability, not business lifecycle authority.
19. End reconciliation with `controller-finalize-cycle` and execute bounded heartbeat cleanup. A pending or failed heartbeat action is returned as `heartbeatAction`/warning, but it must not block failure ingestion, reclaim, recovery, or unrelated registration. Parallel partial-wave delivery remains the only host outbox condition that blocks unrelated batch business, because its member sends are one logical operation.
20. Include generation, automation ID, action ID, RRULE snapshot, occurrence, and fired-at time in scheduled scans. Lost create responses can self-confirm from the matching action ID. An expired pending action returns bounded compensation, but recovery commands stay available and lifecycle truth never advances merely because a host action was prepared.
21. Treat a stale, wrong-ID, expired, `COUNT!=1`, or second occurrence as a host-cleanup path, not a business scan. Return empty work queues plus one identity-scoped `delete_stale_automation`. A second stale trigger or second delete failure opens a persisted fuse and one notification. Permit only one automatic delete compensation; after that return `manual_heartbeat_cleanup_required` with no automatic retry. Never fabricate deletion or rearm from a stale invocation; keep cleanup visible without blocking business lifecycle recovery.
22. Archive only `integrated`, `blocked`, or `reclaimed` tasks. Archive visible descendants first, then their parent. Keep registry and event history; archive only the Codex sidebar thread.
23. After installation or upgrade, run `audit-model-routing`, `audit-thinking-routing`, and `audit-archive-backlog`. The routing audits are read-only; active legacy low-thinking tasks must be reclaimed by their registered direct controller rather than mutated in place. The archive audit groups terminal debt by registered direct controller and labels each item `actionable`, `title_failed`, `archive_failed`, or `waiting_descendants`. Each owner applies only returned actions descendant-first and records success or failure; failed debt stays visible in audit without keeping a heartbeat. Never impersonate another controller or mutate legacy evidence directly.
24. Delete the heartbeat when no dispatched active work, pending failure/review/routing decision, or actionable thread action remains, and also when the stability watchdog fuses despite an active task. Checkpoint creation, checkpoint advice, a prepared/accepted handoff, historical failure/archive debt, a fully notified fuse waiting for user direction, and report freshness never create or retain heartbeat. A fused watchdog pauses automatic monitoring rather than blocking business. Confirm or manually verify cleanup, then use `controller-resume-watchdog --reason ...` only after a controller or user has reviewed the stalled state. Real dispatch/progress/completion resets the no-progress counter.
   Deferred messages are visible in `controller-scan-events`, but do not independently create or retain a heartbeat. Prepared message actions and stale deferred messages require controller attention when a real scan already occurs; their receipts are separate from heartbeat confirmation.

## Safety boundaries

- Do not write `$CODEX_HOME/AGENTS.md`, `$CODEX_HOME/skills/`, project files, or live project ledgers while building or reviewing a candidate.
- Keep project-specific rules in project sources; this skill is an outer control-plane ledger, not a replacement workflow.
- Treat duplicate self-thread IDs across projects, stale events, wrong parents/projects, cycles, invalid enums, and contradictory lifecycle records as errors.
- Reject unsafe thread identifiers before registry reads or event path construction; never interpolate `/`, `\\`, `.`, `..`, or whitespace into an events path.
- Keep provider calls at zero for ledger-only work; this skill records local control evidence and does not invoke a model provider.
- Keep Sol at high by default. Require `audit-controller-routing` evidence before xhigh/max; reject low/none, mechanical escalation, unsupported triggers, missing reasons, and max without explicit authority.
- Forbid internal subagents. Reject any execution surface other than `visible_task`, missing explicit authorization, non-economical workers, low thinking, and missing quota justification.
- Fail closed on a `workClass`/model/thinking mismatch, missing task mode, invalid adaptive brief, unjustified hard-contract escalation, hard-contract digest drift, unresolved product/safety decisions, missing result evidence, controller-only work, non-mechanical redispatch, a second rework, placeholder titles, stale title acknowledgements, partial title/archive metadata, archive-before-terminal, archive-before-title-sync, and parent archive while a visible descendant remains unarchived.
- Never turn hints into a default write-path whitelist, fixed implementation route, fixed stage sequence, or sole validator. Never ask headless execution to prove GUI/screenshots. Preserve contradictory evidence for direct-controller review.
- Fail closed on a missing/drifted result manifest, forged task ownership, artifact path outside controller-approved roots, missing/empty/undecodable visual files, duplicate local hashes, a visual package without presentation evidence, acceptance without a reason, or visual acceptance without a selected image. Do not revalidate old artifact files as lifecycle truth after ingestion; report later disappearance as unavailable while retaining the immutable historical reference.
- Never copy or move project artifacts into task-control. Store normalized references and hashes, render local files through absolute `file:` URIs, and label candidate-worktree, project-main, external-reference, and task-control sources distinctly.
- Never mark a new implementation task integrated from ledger intent alone.
