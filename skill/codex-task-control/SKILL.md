# Codex Task Control

Use the bundled `scripts/task-control.mjs` as the single task-lifecycle and model-routing control plane. It owns visible-task identity, routing, lifecycle evidence, review state, and archive coordination. Project authority remains limited to the project's own development rules, business workflows, commands, tests, and acceptance constraints.

## Workflow

1. Resolve the project root and read its `AGENTS.md` plus the applicable SOP/workflow sources before doing project work.
2. Use the optional project adapter only to locate project rules, applicable business workflows, and project-specific acceptance constraints. Never require or accept a project model-routing source: model, thinking, work-class, and lifecycle policy belong to this user-level skill. Never copy policy text into the adapter or let it override project rules.
3. Set `CODEX_HOME` (or pass `--codex-home`) so the ledger is stored at `$CODEX_HOME/task-control/`; use `--task-control-home` only when passing the exact task-control root. The index is `projects.json`; each project has an isolated `projects/<project-key>/task-registry.json` and `events/` directory.
4. Do the work in the controller by default. Never call internal subagent, multi-agent, or `spawn_agent` capabilities. Delegated work must exist as a user-visible Codex task/thread in the sidebar.
5. Classify work before creating a task shell. Use `repeatable` only when rules, inputs, expected results, and the test oracle are explicit; it must use exactly `gpt-5.6-luna` with medium thinking. Use `bounded_reasoning` only after contracts and policy decisions are resolved but local code understanding is still needed; it must use exactly `gpt-5.6-terra` with medium thinking by default or high thinking when the decided work spans enough code to justify it. Never use low thinking. Keep work in the `gpt-5.6-sol` controller whenever architecture, contract conflict, trusted-source selection, error policy, fixture validity, or another judgment remains unresolved. Route bounded short controller work as `bounded_control` with medium thinking and use `frontier_control` with high thinking by default. Before selecting xhigh, run `audit-controller-routing` with `hard_arbitration`, an allowed escalation trigger, and a concrete reason. Use max only as `final_arbitration` with `user_explicit` or `xhigh_unresolved` authority. Never spend Sol xhigh/max on search, formatting, commands, repeated tests, or mechanical implementation.
6. Authorize a worker only when the expected premium-quota saving exceeds the new context and coordination overhead. A model upgrade never repairs an ambiguous task; resolve decisions and narrow the contract first.
7. Classify every new registration with an explicit `--task-mode`. Use `control_only` only for ledger, audit, inspection, or other work that does not modify project code, resources, UI, tests, or screenshot runners. Use `implementation` for code/resource/test changes and `visual_implementation` when visual output also needs a fixed oracle. Never use `control_only` to bypass an implementation contract. Existing records without this field migrate in memory as `legacy_unclassified`; do not rewrite them merely to silence the distinction.
8. Before registering an implementation task, create a project-owned JSON contract inside the project root. Bind it with `--implementation-contract`; new registrations require schema version 2, `contractRevision` or `contractCommit`, non-empty `allowedWritePaths`, `reuseRequirements`, `forbiddenNewPaths`, `forbiddenReimplementations`, ordered `stageGates`, named `evidenceCommands`, controller-fixed `errorPolicy`, and `resultRequirements`. Existing schema-v1 ledger tasks remain readable and may finish their already-bound attempt, but cannot be newly registered. Result requirements remain schema version 1. A visual task additionally requires `visualOracle`, a required presentation stage, a screenshot/contact-sheet requirement, and an after/current milestone. The skill stores the normalized snapshot and SHA-256 digest, then rechecks it at dispatch, progress, failure, and completion. Any drift fails closed.
9. Before creating task shells, write a project-relative schema-v1 parallel batch manifest and run `controller-plan-parallel-batch`. Record candidates, lanes, dependencies, conflict domains, review cost/capacity, WIP limits, dirty conflict domains, and worktree identity for implementation candidates. Default to at least two independent candidates with distinct incremental value. If the initial wave has only one valuable safe candidate, require a typed `degradationReceipt`; do not add a ceremonial second task. After a real wave starts, completed/blocked/reclaimed peers may naturally shrink the batch to one remaining candidate without a new degradation receipt.
10. Create every task shell needed by the selected fan-out wave only to obtain thread IDs. Register each with `--parallel-policy batch_v1 --batch-id ... --candidate-id ...`, a semantic `--title`, and the existing routing/contract fields. After all selected candidates are registered and their titles are synced, run `controller-prepare-parallel-dispatch`. It atomically records one dispatch wave and returns every required task. Send each real prompt, then call `controller-record-dispatched` for that task. Until every wave member is recorded, unrelated controlled business fails closed and the batch heartbeat keeps the controller accountable. Host message sends cannot be atomic, so an interrupted partial wave remains visible and recoverable rather than being reported as complete.
11. Keep the semantic title stable. The ledger allocates `01`, `02`, and nested `01.1` display keys and changes only the lifecycle prefix: `执行`, `待审`, `待决`, `返工`, `收回`, `接收`, `完成`, or `阻塞`. `changes_requested` means stopped and awaiting a controller routing decision; it never means the worker is running.
12. Visible tasks may be siblings or nested visible tasks. Every visible task must be independently registered and remain directly inspectable by the user. After completion, failure, reclaim, acceptance, integration, or batch closeout, recompute fan-out. When an implementation candidate and a non-conflicting QA/no-code/readonly candidate are both eligible, single-task dispatch is forbidden. Dynamic WIP limits simultaneous review cost and conflict domains, not the total historical task count. Unintegrated candidates cannot satisfy another candidate's dependency.
   Route every later controller-to-worker message through `controller-prepare-message`. Ordinary `follow_up`, `clarification`, `evidence_request`, and `notification` messages use `delivery-mode queue`. When the target is `running` or `unknown`, the command persists `deferred_local` and returns no host send action; do not call `send_message_to_thread`. Re-check the real task turn state and use `controller-release-message` only after it is `idle`; then execute the returned `send_thread_message` and record the matching action ID plus real host receipt. If the task has left `executing`, release cancels the message instead of restarting it. Only `stop` or `cancel` with `user_explicit` or `controller_safety` authority may use `delivery-mode interrupt` and the returned steer action. Never treat the current App setting or a successful generic send call as proof that a running-task message was queued: the current programmatic tool exposes neither an explicit queue mode nor a queue acknowledgement.
13. A worker may only query itself, create progress/completion, submit `report-failure`, or create a notification-failure receipt. For implementation work, an authoritative failure must bind `--evidence-command-id` to the controller-fixed contract. A failure from an ad-hoc or contract-external command is stored as `non_authoritative_diagnostic`; it cannot stop the task, trigger rework, or change acceptance. Use `progress` only for a real checkpoint. A next-stage artifact may be created while exactly one valid predecessor from the same task/attempt/contract is pending ingestion; central state still ingests in order and completion still requires every stage.
14. The controller alone ingests events and changes lifecycle. `controller-dispatch-rework` only prepares a host message and does not increment `attemptCount`; after the host returns a real send receipt, call `controller-confirm-rework-dispatched`. On send failure, cancel the prepared action. Scans surface `zombieAttempts` and `preparedReworks`; `controller-recover-undispatched-attempt` restores a legacy attempt that advanced without a host receipt even when ordinary gates are unhealthy. Objective retry fuses and product-value gates remain unchanged.
15. Piggyback observability on existing lifecycle commands; never add a worker turn merely to measure time. Lean HTML reads only the ledger. Diagnostic HTML is user-requested, local, and may read matching rollout/OTel/Desktop evidence. The report starts with an executive summary that separately counts verified business deliveries, implementation candidates, and control-review results. `control_only` terminal work is “控制审查已通过”, never a business integration. If verified deliverables are zero, say so explicitly. Keep fixed explanations Chinese, use 万/亿 plus exact values, and never assign unknown time or account quota snapshots to one task.
   After a real completion, failure, reclaim, block, or review action, refresh only the lean delivery report. It includes objective runtime/fuse state, failure history, diagnostic value decisions, and incident closeout. `reportNeedsRefresh` alone never keeps heartbeat alive, but an incomplete reclaim/block closeout does. Ingest a schema-v1 context-health receipt from `$codex-time-diagnostics`; `handoff_required` blocks new registration/dispatch until work moves to a clean controller.
16. Use one confirmed `COUNT=1` heartbeat automation per direct controller. Real progress renews only the ledger's logical lease while the confirmed physical automation is still pending; it must not repeatedly replace the App automation. Dispatch, completion, closeout, or an expired physical boundary may prepare a new host action. Heartbeat state and host automation are control-plane observability, not business lifecycle authority.
17. End reconciliation with `controller-finalize-cycle` and execute bounded heartbeat cleanup. A pending or failed heartbeat action is returned as `heartbeatAction`/warning, but it must not block failure ingestion, reclaim, recovery, or unrelated registration. Parallel partial-wave delivery remains the only host outbox condition that blocks unrelated batch business, because its member sends are one logical operation.
18. Include generation, automation ID, action ID, RRULE snapshot, occurrence, and fired-at time in scheduled scans. Lost create responses can self-confirm from the matching action ID. An expired pending action returns bounded compensation, but recovery commands stay available and lifecycle truth never advances merely because a host action was prepared.
19. Treat a stale, wrong-ID, expired, `COUNT!=1`, or second occurrence as a host-cleanup path, not a business scan. Return empty work queues plus an identity-scoped `delete_stale_automation`. Three repeated stale/delete failures open a persisted fuse and one notification. Never fabricate deletion or rearm from a stale invocation; keep cleanup visible without blocking business lifecycle recovery.
20. Archive only `integrated`, `blocked`, or `reclaimed` tasks. Archive visible descendants first, then their parent. Keep registry and event history; archive only the Codex sidebar thread.
21. After installation or upgrade, run `audit-model-routing`, `audit-thinking-routing`, and `audit-archive-backlog`. The routing audits are read-only; active legacy low-thinking tasks must be reclaimed by their registered direct controller rather than mutated in place. The archive audit groups terminal debt by registered direct controller and labels each item `actionable`, `title_failed`, `archive_failed`, or `waiting_descendants`. Each owner applies only returned actions descendant-first and records success or failure; failed debt stays visible in audit without keeping a heartbeat. Never impersonate another controller or mutate legacy evidence directly.
22. Delete the heartbeat only when no dispatched active work, pending failure/review/routing decision, stalled/fused incident requiring closeout, handoff action, or actionable thread action remains. Historical failure/archive debt, a fully notified fuse waiting for user direction, and report freshness alone do not keep it alive. The delete is not complete until its host action is confirmed in the ledger.
   Deferred messages are visible in `controller-scan-events`, but do not independently create or retain a heartbeat. Prepared message actions and stale deferred messages require controller attention when a real scan already occurs; their receipts are separate from heartbeat confirmation.

## Safety boundaries

- Do not write `$CODEX_HOME/AGENTS.md`, `$CODEX_HOME/skills/`, project files, or live project ledgers while building or reviewing a candidate.
- Keep project-specific rules in project sources; this skill is an outer control-plane ledger, not a replacement workflow.
- Treat duplicate self-thread IDs across projects, stale events, wrong parents/projects, cycles, invalid enums, and contradictory lifecycle records as errors.
- Reject unsafe thread identifiers before registry reads or event path construction; never interpolate `/`, `\\`, `.`, `..`, or whitespace into an events path.
- Keep provider calls at zero for ledger-only work; this skill records local control evidence and does not invoke a model provider.
- Keep Sol at high by default. Require `audit-controller-routing` evidence before xhigh/max; reject low/none, mechanical escalation, unsupported triggers, missing reasons, and max without explicit authority.
- Forbid internal subagents. Reject any execution surface other than `visible_task`, missing explicit authorization, non-economical workers, low thinking, and missing quota justification.
- Fail closed on a `workClass`/model/thinking mismatch, missing task mode, implementation work without a complete in-project contract, contract digest drift, missing stage evidence, incomplete required stages, unresolved decisions, missing scope/acceptance/decision boundaries, controller-only work, non-mechanical redispatch, a second rework, placeholder titles, stale title acknowledgements, partial title/archive metadata, archive-before-terminal, archive-before-title-sync, and parent archive while a visible descendant remains unarchived.
- Fail closed on a missing/drifted result manifest, forged task ownership, artifact path outside controller-approved roots, missing/empty/undecodable visual files, duplicate local hashes, a visual package without presentation evidence, acceptance without a reason, or visual acceptance without a selected image. Do not revalidate old artifact files as lifecycle truth after ingestion; report later disappearance as unavailable while retaining the immutable historical reference.
- Never copy or move project artifacts into task-control. Store normalized references and hashes, render local files through absolute `file:` URIs, and label candidate-worktree, project-main, external-reference, and task-control sources distinctly.
- Never mark a new implementation task integrated from ledger intent alone. Resolve the recorded candidate and target ref (default `HEAD`) as Git commits and require the candidate to be an ancestor of the target. Persist the full commits and verification time. Keep legacy integrated records readable, but label missing proof as Git-unverified in reports.
- Never add a worker call, progress event, notification, heartbeat, or provider request only for observability. Default to lean ledger rendering; diagnostic parsing is user-requested, local, read-only, and safe to degrade when rollout/OTel evidence is unavailable. Never present an account quota delta as a per-task bill or assign unknown time to model reasoning, network, or service queueing.
- Never send a work prompt while `dispatchAllowed` is false, and never call `controller-record-dispatched` until the send succeeds. A failed title/archive action is deferred audit debt and must not be re-emitted or keep a heartbeat; only the registered direct controller may explicitly move it back to `pending` with a recorded reason.
- Never dispatch a v1 batch candidate outside a prepared wave. Require distinct incremental value, not a ceremonial candidate count. Initial single-candidate dispatch needs a typed degradation receipt; natural shrink after a real wave does not.
- Never use a repeating heartbeat schedule or in-place heartbeat replacement. Prepare locally, create a new `COUNT=1` automation, confirm its new ID, switch the ledger, and delete the retired ID. A host timeout records failure and preserves the previous confirmed generation.
- Never let heartbeat cleanup mutate or block business lifecycle truth. Keep its host action visible and bounded, but preserve reclaim, recovery, event ingestion, and registration paths. Only a partial parallel dispatch wave remains fail-closed until its real sends are reconciled.
- Never keep a stale invocation silent. Resolve its own automation ID, verify its stored prompt still carries the triggering generation, and execute only the returned bounded `delete_stale_automation`; do not read project context, process queues, or rearm. Treat inability to compare-and-delete atomically as a Codex App limitation and prefer a future host-native stale cleanup hook.
- Never delete terminal task history. `codex_app__set_thread_archived` cleans the sidebar; the ledger retains identity, lifecycle, notification, title-sync, and archive evidence.
- Never treat a completion file as proof of notification. `notificationStatus: sent` requires a real successful message to the registered direct parent.
- Use atomic registry replacement and exclusive event-file creation. Lock files contain `pid`, `createdAt`, and `nonce`; stale recovery requires a threshold check plus repeated owner verification, and release deletes only a matching nonce.
- Run tests with a temporary `--codex-home` or `--task-control-home`; omitted storage arguments must resolve through a test-only `CODEX_HOME` sandbox.
